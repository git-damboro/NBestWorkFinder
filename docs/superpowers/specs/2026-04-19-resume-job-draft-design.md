# 根据简历生成职位草稿设计文档

## 1. 背景

当前项目已经具备以下主链路能力：

| 模块 | 当前能力 |
|---|---|
| `resume` | 上传、解析、分析、查看详情 |
| `job` | 创建职位、编辑、删除、匹配简历、发起定向面试 |
| `interview` | 基于简历或职位定向生成面试、答题、查看记录与报告 |

当前的主要短板不是缺少新模块，而是“从简历进入职位工作台”这一段仍偏手工：

| 当前问题 | 影响 |
|---|---|
| 用户上传简历后，仍需手动新建职位 | 增加输入成本 |
| 职位工作台偏手工录入 | AI 能力感知不强 |
| `resume -> job -> interview` 闭环虽已具备，但起点不够顺滑 | 项目完成度感不够强 |

因此，本次设计目标是增加一个**最小但高价值**的功能：

> 根据简历内容，由 AI 一次生成 3 个适合投递的职位草稿，用户选择 1 个保存到职位工作台。

---

## 2. 目标与非目标

### 2.1 目标

| 目标 | 说明 |
|---|---|
| 降低职位录入门槛 | 用户无需从零填写职位 |
| 提升 AI 感知 | 上传简历后即可看到岗位方向建议 |
| 打通主闭环 | 让 `resume -> job -> match -> interview` 更自然 |
| 保持最小可落地 | 不引入外部招聘源，不做重型新模块 |

### 2.2 非目标

| 不做项 | 原因 |
|---|---|
| 抓取真实招聘网站岗位 | 依赖外部平台，开发与维护成本高 |
| 一次批量保存多个职位草稿 | 会增加交互复杂度，本轮不需要 |
| 自动生成完整真实 JD | 质量不稳定，且容易让草稿失真 |
| 自动填写公司、薪资、地点等真实招聘信息 | 简历通常不具备这些可靠来源 |
| 多轮反复生成 / 精修岗位建议 | 不是当前最小闭环所需 |

---

## 3. 方案选择

### 3.1 候选方案

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| A | 根据简历生成 3 个职位草稿，用户选 1 个保存 | 实现最快、风险最低、最符合当前节奏 | 生成的是“岗位方向草稿”，不是真实职位 |
| B | 根据简历自动预填“新建职位表单” | 改动较小 | 只有 1 个结果，用户选择空间小 |
| C | 根据简历推荐真实招聘岗位 | 产品感最强 | 依赖外部数据源，开发成本明显过高 |

### 3.2 采用方案

本次采用 **方案 A**。

### 3.3 原因

| 原因 | 说明 |
|---|---|
| 快速落地 | 只需在现有 `resume + job` 基础上增强 |
| 用户感知强 | 上传简历后即可看到 AI 推荐的岗位方向 |
| 后续可扩展 | 未来可以升级为批量保存或真实岗位推荐 |
| 与现有链路匹配 | 生成草稿后可直接进入职位工作台，继续匹配与面试 |

---

## 4. 用户流程

### 4.1 主流程

| 步骤 | 用户动作 | 系统行为 |
|---:|---|---|
| 1 | 打开简历详情页 | 页面展示“生成职位草稿”按钮 |
| 2 | 点击按钮 | 前端调用“根据简历生成职位草稿”接口 |
| 3 | 等待结果 | 后端读取简历文本并调用 AI 生成 3 个职位草稿 |
| 4 | 查看草稿 | 前端弹窗展示 3 个草稿卡片 |
| 5 | 选择草稿 | 用户选择其中 1 个并点击保存 |
| 6 | 保存职位 | 前端调用已有 `POST /api/jobs` 创建职位 |
| 7 | 查看结果 | 成功后跳转到新建职位详情页 |

### 4.2 异常流程

| 场景 | 处理方式 |
|---|---|
| 简历不存在或不属于当前用户 | 后端返回权限/不存在错误 |
| 简历还未解析出文本 | 后端返回“简历内容不可用，无法生成职位草稿” |
| AI 生成失败 | 返回统一错误提示，前端显示失败信息 |
| 用户关闭弹窗 | 不保存任何草稿 |
| 用户已生成过但未保存 | 本轮不做历史缓存，重新生成即可 |

---

## 5. 数据设计

### 5.1 职位草稿 DTO

本轮新增一个“职位草稿”响应结构，仅用于前端展示与选择，不直接落库。

| 字段 | 类型 | 说明 |
|---|---|---|
| `title` | `string` | 推荐岗位名称，如“Java 后端开发工程师” |
| `summary` | `string` | 对该岗位方向的简要概括 |
| `reason` | `string` | 推荐原因，解释为什么适合该简历 |
| `techTags` | `string[]` | 从简历中识别出的相关技术标签 |
| `defaultDescription` | `string` | 用于创建职位时填入 `description` 的草稿内容 |
| `defaultNotes` | `string` | 用于创建职位时填入 `notes` 的默认说明 |

### 5.2 草稿保存策略

草稿本身**不单独入库**。  
用户选中草稿后，前端把草稿内容转换成已有 `CreateJobForm` 并调用已有创建职位接口。

### 5.3 创建职位时的默认映射

| 职位字段 | 来源 |
|---|---|
| `title` | 草稿 `title` |
| `company` | 固定默认值：`待补充` |
| `description` | 草稿 `defaultDescription` |
| `notes` | 草稿 `defaultNotes` |
| `applicationStatus` | 默认 `SAVED` |
| `location` | 留空 |
| `salaryMin / salaryMax` | 留空 |

## 6. 接口设计

### 6.1 新增接口

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/api/jobs/drafts/from-resume/{resumeId}` | 根据某份简历生成 3 个职位草稿 |

### 6.2 请求规则

| 项目 | 说明 |
|---|---|
| `resumeId` | 路径参数 |
| 认证 | 必须登录，按当前用户隔离 |
| 请求体 | 无 |

### 6.3 返回结构

```json
[
  {
    "title": "Java 后端开发工程师",
    "summary": "偏向后端系统开发与接口服务建设",
    "reason": "简历中 Java、Spring Boot、MySQL、Redis 经验较集中，适合后端岗位方向",
    "techTags": ["Java", "Spring Boot", "MySQL", "Redis"],
    "defaultDescription": "基于该简历推荐的岗位方向：Java 后端开发工程师。建议重点关注服务开发、数据库设计、缓存应用与系统稳定性建设。",
    "defaultNotes": "该职位由简历智能生成，请补充公司、地点、薪资与具体 JD。"
  }
]
```

### 6.4 保存动作

不新增“保存草稿”接口。  
前端选中后直接调用已有：

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/api/jobs` | 创建真实职位记录 |

## 7. 后端设计

### 7.1 Controller

在 `job` 模块中新增草稿生成接口，由 `job` 控制器对外提供。

### 7.2 Service

新增一个职位草稿生成服务，职责如下：

| 职责 | 说明 |
|---|---|
| 校验简历归属 | 使用 `resumeId + userId` 查找简历 |
| 校验简历文本 | 确保 `resumeText` 存在且非空 |
| 调用 AI | 基于简历生成 3 个职位草稿 |
| 结果清洗 | 去空、去重、补默认文案 |
| 返回 DTO | 供前端展示与选择 |

### 7.3 AI Prompt 规则

提示词应约束模型：

| 规则 | 说明 |
|---|---|
| 只生成 3 个草稿 | 避免结果太多 |
| 必须基于简历真实技能与经历 | 不得推荐完全无关岗位 |
| 岗位名称要常见、易理解 | 避免生成怪异职位名 |
| 总结与理由要简洁 | 方便前端卡片展示 |
| 不输出薪资、公司、地点等不可靠信息 | 避免误导用户 |

### 7.4 安全与隔离

| 点 | 处理 |
|---|---|
| 用户隔离 | 必须按 `resumeId + userId` 读取简历 |
| 越权访问 | 非本人简历不可生成草稿 |
| 结果落库 | 草稿不落库，只有用户确认保存后才创建职位 |

## 8. 前端设计

### 8.1 入口位置

入口放在 **简历详情页**。

理由如下：

| 原因 | 说明 |
|---|---|
| 语义最自然 | “根据这份简历生成职位草稿” |
| 依赖最明确 | 用户已经进入具体简历详情，`resumeId` 明确 |
| 改动最集中 | 不需要大范围改导航与其他页面 |

### 8.2 前端组件

建议新增一个弹窗组件，例如：

| 组件 | 用途 |
|---|---|
| `ResumeJobDraftDialog` | 展示 3 个职位草稿，并允许用户选择 1 个保存 |

### 8.3 弹窗内容

每个草稿卡片展示：

| 展示项 | 内容 |
|---|---|
| 岗位名称 | `title` |
| 岗位概述 | `summary` |
| 推荐原因 | `reason` |
| 技术标签 | `techTags` |
| 操作按钮 | “保存到职位工作台” |

### 8.4 保存后的跳转

用户选择并保存成功后：

| 动作 | 结果 |
|---|---|
| 创建职位成功 | 关闭弹窗 |
| 页面跳转 | 进入新建职位详情页 |

采用跳转到职位详情页，而不是仅刷新当前页，原因如下：

| 原因 | 说明 |
|---|---|
| 用户更容易继续补全信息 | 公司、地点、薪资、备注可立即完善 |
| 更符合职位工作流 | 后续可以直接继续匹配与定向面试 |

---

## 9. 错误处理

| 场景 | 前端提示 |
|---|---|
| 简历未解析完成 | “当前简历内容尚不可用，请先完成解析或分析后再试。” |
| AI 生成失败 | “职位草稿生成失败，请稍后重试。” |
| 保存职位失败 | “保存职位失败，请稍后重试。” |
| 返回草稿为空 | “未生成可用职位草稿，请尝试更新简历内容后再试。” |

## 10. 测试策略

### 10.1 后端测试

| 测试点 | 说明 |
|---|---|
| 用户隔离测试 | 只能基于当前用户自己的简历生成草稿 |
| 简历文本为空 | 正确返回业务错误 |
| AI 正常返回 | 能输出 3 个规范草稿 |
| AI 异常返回 | 能正确抛出或包装错误 |

### 10.2 前端验证

| 验证点 | 说明 |
|---|---|
| 简历详情页入口显示正常 | 按钮位置和状态正确 |
| 弹窗展示 3 个草稿 | 卡片内容显示正常 |
| 选择草稿后能成功创建职位 | 并跳转到职位详情 |
| 失败提示正常 | 生成失败、保存失败都有清晰反馈 |

## 12. 结论

本设计选择以**最小闭环**的方式增强项目完成度：

| 结论 | 内容 |
|---|---|
| 入口 | 简历详情页 |
| 输出 | 3 个职位草稿 |
| 操作 | 用户选择 1 个保存 |
| 保存结果 | 创建为真实职位并进入职位详情页 |
| 核心价值 | 让“简历 → 职位”从手工录入升级为 AI 辅助生成 |

这会是一个开发成本较低、用户感知明显、且能快速提升项目完整度的功能点。
