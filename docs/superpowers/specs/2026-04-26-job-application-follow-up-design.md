# 辅助投递闭环设计

## 1. 背景

当前系统已经具备职位收集和职位处理的基础能力：

| 能力 | 状态 |
|---|---|
| 浏览器扩展同步 BOSS 列表职位 | 已有 |
| BOSS 详情页补全 JD | 已有 |
| 职位草稿池筛选、编辑、导入 | 已有 |
| 职位工作台状态管理 | 已有 |
| 简历匹配 | 已有 |
| 定向面试 | 已有 |

但“辅助投递”仍缺少真正的投递跟进闭环。当前职位只有一个 `applicationStatus`，用户无法清晰记录投递时间、沟通内容、下一步跟进时间，也无法在职位详情中回顾完整时间线。

本设计目标是补齐系统内的投递跟进能力，让职位从“收藏/草稿”进入“投递、沟通、面试、结果”的完整流程。

## 2. 范围

本轮采用已确认的 C 方案：

| 项 | 内容 |
|---|---|
| 状态变化自动记录 | 每次职位投递状态变化自动写入一条跟进记录 |
| 手动跟进日志 | 用户可手动添加沟通、面试、备注、Offer、拒绝等记录 |
| 职位快照字段 | 在职位上保存投递时间、最近跟进时间、下一步跟进时间 |
| 前端时间线 | 职位详情弹窗展示完整投递跟进时间线 |
| 快捷动作 | 支持在职位详情里快速标记已投递、面试中、Offer、拒绝 |

本轮暂不做：

| 暂不做 | 原因 |
|---|---|
| 独立联系人/公司 CRM 模块 | 当前阶段过重，会拖慢投递闭环 |
| 跟进提醒通知/日历 | 需要额外通知机制，后续单独做 |
| 删除跟进记录 | 避免误删历史，本轮先保留完整记录 |
| 扩展内直接标记投递 | BOSS DOM 风险较高，先把核心闭环放在系统内 |

## 3. 用户流程

| 阶段 | 用户动作 | 系统表现 |
|---|---|---|
| 收集职位 | BOSS 同步、简历生成、手动新增 | 职位进入草稿池或职位工作台 |
| 准备投递 | 查看 JD、匹配分、开场话术 | 用户决定是否投递 |
| 执行投递 | 点击“标记已投递” | 职位变为已投递，写入投递时间线 |
| 后续沟通 | 添加跟进记录 | 时间线记录沟通方式、内容和下一步时间 |
| 状态推进 | 改为面试中、Offer、拒绝 | 自动写入状态变化记录 |
| 回顾管理 | 打开职位详情 | 展示完整时间线、最近跟进和下一步跟进 |

## 4. 后端设计

### 4.1 数据模型

新增表：`job_follow_up_records`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint | 主键 |
| `job_id` | bigint | 职位 ID |
| `user_id` | bigint | 用户 ID，用于隔离 |
| `type` | varchar | 跟进类型 |
| `title` | varchar | 记录标题 |
| `content` | text | 记录内容 |
| `from_status` | varchar | 状态变化前状态 |
| `to_status` | varchar | 状态变化后状态 |
| `contact_method` | varchar | 沟通方式 |
| `next_follow_up_at` | timestamp | 下一步跟进时间 |
| `created_at` | timestamp | 创建时间 |

新增枚举：`JobFollowUpType`

| 类型 | 说明 |
|---|---|
| `STATUS_CHANGE` | 状态变化自动记录 |
| `MANUAL_NOTE` | 手动备注 |
| `CONTACT` | 沟通记录 |
| `INTERVIEW` | 面试记录 |
| `OFFER` | Offer 记录 |
| `REJECTION` | 拒绝/不合适记录 |

扩展 `jobs` 表：

| 字段 | 说明 |
|---|---|
| `applied_at` | 首次进入已投递状态的时间 |
| `last_follow_up_at` | 最近一次跟进记录时间 |
| `next_follow_up_at` | 下一步跟进时间 |

### 4.2 服务边界

新增 `JobFollowUpService`，职责如下：

| 方法 | 说明 |
|---|---|
| `list(jobId, userId)` | 查询当前用户某职位的跟进记录 |
| `createManual(jobId, request, userId)` | 手动新增跟进记录 |
| `recordStatusChange(job, fromStatus, toStatus)` | 状态变化时自动记录 |

`JobService.update` 负责检测状态变化：

1. 加载当前用户职位。
2. 记录旧 `applicationStatus`。
3. 应用更新请求。
4. 如果状态发生变化，调用 `JobFollowUpService.recordStatusChange`。
5. 如果新状态首次进入 `APPLIED`，写入 `appliedAt`。
6. 保存职位并返回 DTO。

### 4.3 API

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/jobs/{id}/follow-ups` | 查询职位跟进时间线 |
| `POST` | `/api/jobs/{id}/follow-ups` | 新增手动跟进记录 |

请求 DTO：`CreateJobFollowUpRequest`

| 字段 | 说明 |
|---|---|
| `type` | 跟进类型，默认可为 `MANUAL_NOTE` |
| `title` | 标题，可空，后端可按类型补默认标题 |
| `content` | 内容 |
| `contactMethod` | 沟通方式，例如 BOSS、微信、电话、邮件 |
| `nextFollowUpAt` | 下一步跟进时间，可空 |

响应 DTO：`JobFollowUpRecordDTO`

| 字段 | 说明 |
|---|---|
| `id` | 记录 ID |
| `jobId` | 职位 ID |
| `type` | 类型 |
| `title` | 标题 |
| `content` | 内容 |
| `fromStatus` | 变化前状态 |
| `toStatus` | 变化后状态 |
| `contactMethod` | 沟通方式 |
| `nextFollowUpAt` | 下一步跟进时间 |
| `createdAt` | 创建时间 |

## 5. 前端设计

### 5.1 类型和 API

新增：

| 文件 | 内容 |
|---|---|
| `frontend/src/types/job-follow-up.ts` | 跟进记录类型、请求类型、标签映射 |
| `frontend/src/api/jobFollowUps.ts` | 查询和新增职位跟进记录 API |

扩展 `JobDetail`：

| 字段 | 说明 |
|---|---|
| `appliedAt` | 投递时间 |
| `lastFollowUpAt` | 最近跟进时间 |
| `nextFollowUpAt` | 下一步跟进时间 |

### 5.2 职位详情弹窗

在 `JobDetailModal` 中新增“投递跟进时间线”区域：

| 状态 | 展示 |
|---|---|
| 加载中 | 显示加载态 |
| 无记录 | 显示空态，引导标记投递或添加备注 |
| 有记录 | 按创建时间倒序展示类型、标题、内容、时间、下一步跟进 |

新增快捷状态动作：

| 按钮 | 行为 |
|---|---|
| 标记已投递 | 调用 `updateJob` 设置 `APPLIED` |
| 进入面试中 | 调用 `updateJob` 设置 `INTERVIEWING` |
| 标记 Offer | 调用 `updateJob` 设置 `OFFERED` |
| 标记拒绝 | 调用 `updateJob` 设置 `REJECTED` |

### 5.3 添加跟进记录弹窗

新增组件：`JobFollowUpDialog`

字段：

| 字段 | 控件 |
|---|---|
| 类型 | 下拉选择 |
| 沟通方式 | 输入框或下拉 |
| 内容 | 多行文本 |
| 下一步跟进时间 | 日期时间输入 |

保存后：

1. 关闭弹窗。
2. 刷新跟进时间线。
3. 刷新职位详情快照字段。

## 6. 错误处理

| 场景 | 处理 |
|---|---|
| 职位不存在 | 后端返回 `JOB_NOT_FOUND` |
| 跨用户访问 | 与不存在同样返回不可访问错误 |
| 内容为空 | 前端校验，后端也校验 |
| 状态未变化 | 不生成重复状态变化记录 |
| 手动记录保存失败 | 弹窗保留输入，显示错误 |
| 时间线加载失败 | 职位详情仍可显示，时间线区域显示重试 |

## 9. 验收标准

| 编号 | 标准 |
|---|---|
| 1 | 职位从已收藏改为已投递后，自动生成状态变化记录 |
| 2 | 再次保存相同状态，不重复生成状态变化记录 |
| 3 | 用户可手动添加沟通/备注记录 |
| 4 | 手动记录能保存下一步跟进时间 |
| 5 | 职位详情能展示完整时间线 |
| 6 | 职位详情能看到投递时间、最近跟进和下一步跟进 |
| 7 | 跨用户不能查看或写入他人职位跟进 |
| 8 | 草稿导入职位后仍默认已收藏，不破坏现有导入链路 |
