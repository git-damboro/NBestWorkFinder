# BOSS 当前页职位同步与统一职位草稿池设计

| 项 | 内容 |
|---|---|
| 文档日期 | 2026-04-20 |
| 主题 | BOSS 当前页职位同步 + 统一职位草稿池 |
| 适用范围 | 简历生成职位草稿、BOSS 当前页同步职位草稿、职位工作台承接 |
| 当前阶段 | 设计完成，待实现计划 |
| 设计目标 | 用最低用户操作成本，把外部职位浏览行为与系统内排序、补全、分析、保存流程打通 |

## 1. 背景与问题

当前系统已经具备单职位保存、职位匹配分析、职位工作台承接等能力，但还缺少一层“批量职位草稿承接层”，因此存在以下问题：

| 问题 | 现状 |
|---|---|
| 外部职位导入成本高 | 用户如果要一个个复制 JD 或逐条导入，体验太差，容易放弃系统 |
| 职位推荐入口不统一 | “简历生成职位草稿”和“外部同步职位草稿”没有统一承接层 |
| 切页后任务容易丢感知 | 面试题、职位草稿、分析任务如果与页面绑定，用户切页后容易觉得任务丢失 |
| 列表页与详情页职责不清 | 当前没有明确区分“快速粗排序”和“完整 JD 精分析” |

因此，需要增加一套统一的“职位草稿池 + 批次管理 + 后台异步分析 + 系统页恢复”方案。

## 2. 目标与非目标

### 2.1 目标

| 目标 | 说明 |
|---|---|
| 降低导入成本 | 用户在 BOSS 列表页点击一次即可同步当前页可见职位 |
| 统一草稿承接 | 简历生成岗位推荐与 BOSS 同步岗位统一进入职位草稿池 |
| 支持批量确认 | 用户可多选职位草稿，一次性保存到职位工作台 |
| 分层分析 | 列表页先粗匹配，详情页补全后再精匹配 |
| 后台持续运行 | 分析、补全、生成过程不依赖前端页面持续停留 |
| 可恢复 | 用户切走页面、刷新页面、重新打开系统后，都能恢复最近批次 |

### 2.2 非目标

| 非目标 | 说明 |
|---|---|
| 全自动投递 | 本期不实现自动开聊、自动发送、自动投递 |
| 全站爬取 | 本期不做后台无感批量爬整站职位 |
| 扩展侧重操作台 | 第一版不把复杂筛选、排序、确认逻辑放在浏览器扩展中 |
| 直接写入正式职位表 | 当前页同步职位不会绕过草稿池直接进入 `jobs` |

## 3. 用户主流程

系统采用“系统页为主、扩展抓取为辅”的闭环。

| 步骤 | 用户动作 | 系统动作 |
|---:|---|---|
| 1 | 用户打开系统并准备简历上下文 | 系统记录当前默认简历 |
| 2 | 用户打开 BOSS 列表页浏览职位 | 扩展识别当前页可见职位卡片 |
| 3 | 用户点击“同步当前页职位” | 扩展提交当前页职位数据到后端，创建草稿批次 |
| 4 | 用户切回系统 | 系统根据 `batchId` 打开职位草稿面板 |
| 5 | 用户查看排序结果 | 系统展示粗匹配分、摘要、待补全状态 |
| 6 | 用户决定深入某个职位 | 用户打开该职位详情页，扩展尝试补全完整 JD |
| 7 | 用户返回系统继续看结果 | 系统展示精匹配分、开场话术等更完整结果 |
| 8 | 用户多选确认 | 系统批量保存到职位工作台 |
| 9 | 用户后续继续管理 | 正式职位进入职位工作台后参与后续流程 |

## 4. 设计总览

### 4.1 总体分层

| 层 | 职责 |
|---|---|
| 浏览器扩展同步层 | 在 BOSS 列表页与详情页提取职位信息、提交系统 |
| 职位草稿承接层 | 接收各种来源的职位草稿，形成可恢复批次 |
| 排序分析层 | 对职位做粗匹配与精匹配分析 |
| 草稿选择面板层 | 向用户展示批次、状态、排序、选择结果 |
| 正式职位层 | 把用户确认过的职位保存到职位工作台 |
| 后台任务层 | 承接耗时分析与补全任务，保证切页后任务继续运行 |

### 4.2 设计原则

| 原则 | 说明 |
|---|---|
| 快慢分层 | 列表页负责快，详情页负责准 |
| 统一承接 | 不同来源职位先入草稿池，再入正式职位 |
| 状态透明 | 用户能明确看到已导入、已勾选、待补全、分析中等状态 |
| 后台异步 | 页面只是发起和查看任务，任务执行不依赖页面停留 |
| 最少打扰 | 用户只保持自然浏览流程，不要求额外复杂导入操作 |

## 6. 接口设计

### 6.1 草稿批次创建

| 接口 | 用途 |
|---|---|
| `POST /api/job-drafts/batches/from-resume/{resumeId}` | 从“简历生成职位草稿”创建批次 |
| `POST /api/job-drafts/batches/from-page-sync` | 从浏览器扩展同步当前页职位创建批次 |

`POST /api/job-drafts/batches/from-page-sync` 请求体建议：

| 字段 | 说明 |
|---|---|
| `resumeId` | 当前使用简历 |
| `sourcePlatform` | 来源平台，如 `BOSS` |
| `sourcePageUrl` | 当前列表页 URL |
| `sourcePageTitle` | 当前列表页标题 |
| `pageFingerprint` | 当前页面指纹 |
| `jobs[]` | 当前页可见职位卡片数组 |

`jobs[]` 子项建议字段：

| 字段 | 说明 |
|---|---|
| `externalJobId` | 外部职位 ID |
| `sourceUrl` | 原始职位链接 |
| `title` | 职位名 |
| `company` | 公司名 |
| `location` | 地点 |
| `salaryTextRaw` | 原始薪资文案 |
| `salaryMin` | 薪资下限 |
| `salaryMax` | 薪资上限 |
| `experienceTextRaw` | 经验文案 |
| `educationTextRaw` | 学历文案 |
| `descriptionPreview` | 列表摘要 |
| `techTags` | 技术标签 |
| `benefits` | 福利标签 |
| `recruiterName` | 招聘方姓名 |
| `rawPayload` | 原始抓取载荷 |

返回体建议：

| 字段 | 说明 |
|---|---|
| `batchId` | 新建批次 ID |
| `status` | 当前批次状态 |
| `totalCount` | 同步数量 |
| `resumeId` | 使用的简历 ID |
| `taskId` | 后台分析任务 ID |
| `needResumeSelection` | 是否还需要前端补充简历选择 |

### 6.2 批次查询与恢复

| 接口 | 用途 |
|---|---|
| `GET /api/job-drafts/batches/{batchId}` | 查询批次摘要 |
| `GET /api/job-drafts/batches/{batchId}/items` | 查询批次草稿项列表 |
| `GET /api/job-drafts/batches/latest` | 查询最近可恢复批次 |

### 6.3 选择状态维护

| 接口 | 用途 |
|---|---|
| `PUT /api/job-drafts/batches/{batchId}/selection` | 全量覆盖当前批次勾选项 |
| `PUT /api/job-drafts/batches/{batchId}/resume` | 切换当前批次简历并触发重排 |

说明：`selection` 接口采用“本批次勾选状态整体覆盖”的方式，前端维护当前勾选集合，后端一次性覆盖保存，避免增量操作带来的状态漂移。

### 6.4 批量保存到职位工作台

| 接口 | 用途 |
|---|---|
| `POST /api/job-drafts/batches/{batchId}/import` | 把选中的草稿项批量导入职位工作台 |

导入行为约束：

| 规则 | 说明 |
|---|---|
| 已导入项跳过 | 避免重复写入正式职位 |
| 批量创建正式职位 | 保留来源信息与草稿引用 |
| 回写导入结果 | 更新 `is_imported` 和 `imported_job_id` |
| 更新批次统计 | 更新 `selected_count`、`imported_count`、`status` |

### 6.5 详情页补全与分析

| 接口 | 用途 |
|---|---|
| `POST /api/job-drafts/items/{draftItemId}/detail-sync` | 对单条草稿做详情 JD 补全 |
| `POST /api/jobs/{id}/detail-sync` | 对正式职位做详情补全 |

### 6.6 分析层分工

| 分析类型 | 输入 | 输出 |
|---|---|---|
| 粗匹配 | 列表页卡片信息 + 简历 | `coarse_match_score` |
| 精匹配 | 完整 JD + 简历 | `precise_match_score`、`match_summary`、`opener_text` |

## 7. 前端系统页设计

### 7.1 页面定位

建议在职位工作台下新增固定入口：`职位草稿`。

| 项 | 设计 |
|---|---|
| 页面路由 | `/jobs/drafts?batchId=xxx` |
| 页面职责 | 承接所有草稿批次、展示排序、支持批量保存 |
| 页面形态 | 列表主区 + 右侧详情抽屉 |

### 7.2 页面结构

| 区块 | 内容 | 作用 |
|---|---|---|
| 批次头部卡片 | 来源、简历、总数、已选、已导入、状态、最近更新时间 | 让用户看到当前批次整体情况 |
| 筛选排序栏 | 来源、导入状态、勾选状态、JD 完整度、匹配分过滤 | 快速缩小候选范围 |
| 草稿列表区 | 职位草稿卡片或表格 | 支持多选与快速浏览 |
| 详情抽屉 | JD 摘要、完整 JD、匹配说明、开场话术 | 不跳页看详情 |
| 底部批量操作栏 | 批量保存、取消勾选、重试补全 | 集中处理多选项 |

### 7.3 草稿列表字段

| 字段 | 是否必显 | 说明 |
|---|---:|---|
| 勾选框 | 是 | 支持多选 |
| 职位名 / 公司名 | 是 | 基础识别信息 |
| 城市 / 薪资 / 经验 / 学历 | 是 | 列表快速判断 |
| 来源标签 | 是 | `BOSS同步`、`简历生成` |
| 导入状态 | 是 | `已导入` / `未导入` |
| 勾选状态 | 是 | 当前批次是否已勾选 |
| JD 完整度 | 是 | `待补全` / `部分补全` / `已完整` |
| 粗匹配分 | 是 | 列表页粗排序结果 |
| 精匹配分 | 条件显示 | JD 完整后显示 |
| 匹配摘要 | 建议 | 帮助用户快速理解 |
| 开场话术 | 条件显示 | 精分析后可直接参考 |
| 更新时间 | 建议 | 判断后台是否还在处理 |

### 7.4 状态展示

| 状态类型 | 值 | 展示建议 |
|---|---|---|
| 批次状态 | `CREATED` | 已创建，待分析 |
| 批次状态 | `ANALYZING` | 顶部进度条 + 正在粗排序 |
| 批次状态 | `READY` | 可正常勾选与批量保存 |
| 批次状态 | `PARTIAL_IMPORTED` | 顶部提示部分已导入 |
| 批次状态 | `COMPLETED` | 顶部提示本批次已处理完成 |
| 批次状态 | `FAILED` | 顶部错误提醒，可重试 |
| 详情同步状态 | `UNSYNCED` | 待补全 |
| 详情同步状态 | `PARTIAL` | 部分补全 |
| 详情同步状态 | `COMPLETED` | JD 已完整 |
| 详情同步状态 | `FAILED` | 补全失败，可重试 |

## 8. 浏览器扩展交互设计

### 8.1 列表页同步

| 场景 | 扩展行为 | 用户感知 |
|---|---|---|
| 进入 BOSS 列表页 | 自动识别当前页可见职位卡片数量 | 弹窗显示检测数量 |
| 点击同步 | 提取当前页可见职位并提交后端 | 显示“正在同步” |
| 同步成功 | 拿到 `batchId` | 提供“打开系统查看”按钮 |
| 用户继续浏览 | 无需停留扩展弹窗 | 后端继续分析 |

### 8.2 详情页补全

| 场景 | 扩展行为 | 用户感知 |
|---|---|---|
| 打开职位详情页 | 自动识别职位是否属于已同步草稿 | 无感触发补全 |
| 抓取成功 | 回传完整 JD、补全状态 | 系统页自动显示更完整结果 |
| 抓取失败 | 判断是否因“展开全文”缺失导致 | 提示用户展开后重试 |

### 8.3 弹窗最小化设计

第一版扩展弹窗保持极简，只保留最小闭环操作。

| 模块 | 内容 |
|---|---|
| 当前状态区 | 当前账号、默认简历、当前页识别职位数 |
| 主按钮 | `同步当前页职位` |
| 次按钮 | `打开系统草稿面板` |
| 最近结果 | 最近一次同步时间、数量、状态 |
| 说明 | 列表页负责快排序，详情页负责精补全 |

## 9. 后台异步任务设计

### 9.1 核心原则

页面不直接“持有任务”，页面只发起任务并查看状态；真正执行发生在后端异步任务层。

| 项 | 设计 |
|---|---|
| 任务归属 | 属于用户与批次，不属于某个前端页面实例 |
| 页面切走 | 任务继续执行 |
| 页面回来 | 根据 `batchId` 或最近批次恢复 |
| 首期通知方式 | 轮询 |
| 后续优化 | 可升级为 SSE / WebSocket |

### 9.2 可复用价值

这套任务机制不仅可用于职位草稿分析，也可复用到以下能力：

| 能力 | 可复用内容 |
|---|---|
| 面试题生成 | 后台任务创建、状态轮询、页面恢复 |
| 职位草稿生成 | 异步生成与分页恢复 |
| 开场话术生成 | 任务状态统一管理 |
| JD 补全分析 | 补全后自动触发精分析 |

## 10. 页面恢复逻辑

| 场景 | 恢复方式 |
|---|---|
| 扩展刚同步完成 | 直接打开 `/jobs/drafts?batchId=xxx` |
| 用户刷新 | 从 URL 参数恢复当前批次 |
| 用户稍后回来 | 从 `GET /api/job-drafts/batches/latest` 恢复最近批次 |
| 用户存在多个批次 | 页面顶部提供批次切换 |
| 批次已完成 | 仍可查看历史，但默认优先打开最近进行中的批次 |

## 11. 与“简历生成职位草稿”的统一改造

“简历生成职位草稿”不再走独立一次性结果页，而是与 BOSS 当前页同步结果统一进入同一草稿池和同一草稿面板。

| 原流程 | 改造后 |
|---|---|
| 生成后展示独立结果页 | 进入统一 `职位草稿` 页面 |
| 可能偏单条确认 | 支持多选 |
| 不一定记录已勾选状态 | 显式保存 `is_selected` |
| 已导入状态不明显 | 清晰展示 `已导入` |
| 来源之间体验割裂 | 不同来源同一交互模型 |

## 12. 重复职位与状态策略

| 场景 | 处理策略 |
|---|---|
| 当前页同步到已导入职位 | 默认不勾选，但明确展示“已导入” |
| 同一职位在多个批次重复出现 | 允许保留草稿记录，但导入时跳过已入正式职位项 |
| 用户本次已勾选 | 保持勾选状态可恢复 |
| JD 自动补全失败 | 标记失败并提示用户展开全文或手动重试 |

## 13. 性能与时效预期

| 环节 | 目标耗时 | 说明 |
|---|---|---|
| 当前页同步 + 粗排序 | 2~8 秒 | 取决于当前页职位数量与分析队列压力 |
| 详情页 JD 补全 + 精分析 | 3~12 秒 | 取决于详情抓取完整度与模型处理耗时 |

说明：列表页负责“先给结果”，详情页负责“结果更准”，不要求用户先完整浏览所有职位。

## 15. 风险与约束

| 风险/约束 | 说明 | 应对 |
|---|---|---|
| 页面 DOM 易变化 | BOSS 页面结构可能调整 | 抽象扩展采集层，允许快速适配选择器 |
| 当前页信息不完整 | 列表页拿不到完整 JD | 采用“粗匹配 + 详情补全”双阶段 |
| 用户切页造成误解 | 用户会误以为任务中断 | 明确任务状态、最近批次恢复、后台异步化 |
| 重复职位较多 | 多批次可能出现重复 | 统一指纹去重，导入阶段跳过重复正式职位 |

## 16. 最终结论

本设计采用以下确定方案：

| 项 | 最终选择 |
|---|---|
| 产品主模式 | 系统页为主 |
| 外部职位同步方式 | 浏览器扩展同步当前页可见职位 |
| 承接模型 | 统一职位草稿池 |
| 用户确认方式 | 草稿面板多选后一次性保存到职位工作台 |
| 重复职位处理 | 默认不勾选，明确显示已导入 |
| JD 补全策略 | 自动补全优先，失败后提示用户辅助操作 |
| 长任务策略 | 后台异步执行，前端负责恢复与展示 |

该方案能在不走高风险自动投递路线的前提下，最大化降低用户导入成本，并补齐系统从“发现职位”到“分析职位”再到“保存正式职位”的关键闭环。
