# 任务中心 V1 设计文档

| 项 | 内容 |
|---|---|
| 文档日期 | 2026-04-20 |
| 主题 | 统一任务中心 V1 |
| 适用范围 | 职位草稿生成、面试会话生成、BOSS 当前页同步、JD 补全 |
| 当前阶段 | 设计确认完成，待进入实现计划 |
| 设计目标 | 为系统补齐“统一任务控制台”，让长任务可见、可恢复、可重试、可跳转结果 |

## 1. 背景与问题

当前项目已经具备多条异步/长任务链路，但任务状态分散在各业务页面中，缺少统一控制台。

| 现状 | 问题 |
|---|---|
| 简历页可发起“生成职位草稿”任务 | 用户切页后不容易重新找到任务 |
| 面试页可发起“生成面试会话/题目”任务 | 任务状态只在原页面感知，恢复入口分散 |
| 浏览器扩展可发起“同步当前页职位”“补全当前 JD” | 当前主要由扩展本地状态感知，系统内缺少统一记录 |
| 各页面各自轮询或局部恢复 | 代码分散，用户体验不统一 |

因此，V1 需要新增一个统一任务中心页面，并把现有 AI 任务与扩展/JD 补全任务一起纳入同一套任务记录与查看入口。

## 2. 目标与非目标

### 2.1 目标

| 目标 | 说明 |
|---|---|
| 统一展示任务 | 一个页面展示所有长任务 |
| 支持切页恢复 | 用户离开原页面后仍可查看任务状态 |
| 支持失败重试 | 失败任务可在任务中心直接重试 |
| 支持结果跳转 | 完成任务可直接跳到对应结果页 |
| 统一任务类型 | 让 AI 任务、扩展同步、JD 补全都进入统一任务记录 |
| 最小侵入改造 | 尽量复用现有 `AiGenerationTask` 基础能力，不重做整套任务系统 |

### 2.2 非目标

| 非目标 | 说明 |
|---|---|
| 批量操作 | V1 不支持多选任务后一键重试/删除/取消 |
| 通知中心 | V1 不做站内通知、角标提醒、浏览器通知 |
| 任务取消 | V1 不支持取消运行中的任务 |
| 复杂分页统计 | V1 不做复杂报表，只做必要统计卡片与筛选 |
| 全量历史归档治理 | V1 不做任务清理策略页面 |

## 3. V1 范围

### 3.1 纳入任务中心的任务类型

| 任务类型 | 当前状态 | V1 处理方式 |
|---|---|---|
| `RESUME_JOB_DRAFT` | 已存在 | 直接纳入任务中心 |
| `INTERVIEW_SESSION_CREATE` | 已存在 | 直接纳入任务中心 |
| `JOB_DRAFT_PAGE_SYNC` | 新增 | 承接扩展“同步当前页职位” |
| `JOB_DRAFT_DETAIL_SYNC` | 新增 | 承接扩展/系统“补全当前 JD” |

### 3.2 暂不纳入

| 动作 | 原因 |
|---|---|
| 批量导入职位工作台 | 短动作，通常瞬时完成，不属于长任务核心场景 |
| 职位匹配分析即时动作 | 当前不是独立长任务，不强制纳入 V1 |

## 4. 总体方案

### 4.1 方案选择

| 方案 | 描述 | 结论 |
|---|---|---|
| 纯前端聚合 | 页面拼接 AI 任务 + 扩展本地状态 | 不选，状态源分散 |
| 统一任务表承接 | 全部任务统一落到 `AiGenerationTask` 体系 | 选用 |
| 双轨任务系统 | AI 任务一套、扩展任务另一套 | 不选，后续维护成本高 |

### 4.2 总体设计原则

| 原则 | 说明 |
|---|---|
| 一个页面统一看 | 用户只记住“任务中心”这一个入口 |
| 一个任务一个结果入口 | 每个任务都能明确“完成后去哪里” |
| 一个任务一个重试动作 | 失败后可以单个重试，避免复杂批量语义 |
| 统一后端记录 | 浏览器扩展不再只依赖本地状态 |
| 先补通路再做体验增强 | V1 先打通记录、展示、跳转、重试 |

## 5. 数据模型设计

### 5.1 复用现有任务实体

V1 优先复用现有 `AiGenerationTaskEntity` / `AiGenerationTaskDTO`，通过扩展任务类型和结果字段承接更多任务，而不是新建第二套任务表。

### 5.2 任务类型扩展

| 枚举 | 含义 |
|---|---|
| `RESUME_JOB_DRAFT` | 根据简历生成职位草稿 |
| `INTERVIEW_SESSION_CREATE` | 生成面试会话/题目 |
| `JOB_DRAFT_PAGE_SYNC` | 浏览器扩展同步当前页职位到草稿池 |
| `JOB_DRAFT_DETAIL_SYNC` | 对草稿项或正式职位执行 JD 补全 |

### 5.3 结果字段约定

V1 不额外拆表，统一通过 `resultJson` 承接任务结果上下文。

| 任务类型 | `resultJson` 关键字段 | 作用 |
|---|---|---|
| `RESUME_JOB_DRAFT` | `batchId`、`resumeId` | 跳转职位草稿页 |
| `INTERVIEW_SESSION_CREATE` | `sessionId`、`resumeId` | 跳转面试页/面试记录 |
| `JOB_DRAFT_PAGE_SYNC` | `batchId`、`resumeId`、`sourcePlatform` | 跳转职位草稿页 |
| `JOB_DRAFT_DETAIL_SYNC` | `batchId`、`draftItemId`、`jobId`、`targetKind` | 跳转职位草稿页或职位工作台 |

### 5.4 来源字段约定

| 字段 | 用法 |
|---|---|
| `sourceId` | 统一存放“源对象 ID”，如 `resumeId`、`batch` 关联主对象 ID |
| `targetId` | 统一存放目标对象 ID，如 `jobId`、`interview session` 关联对象 |

若当前 `sourceId/targetId` 不能完整表达扩展任务上下文，则以 `resultJson` 为准，不在 V1 额外重构通用模型。

## 6. 后端设计

### 6.1 新增/扩展接口

| 接口 | 说明 |
|---|---|
| `GET /api/ai-generation/tasks` | 任务中心列表接口，按用户查询最近任务 |
| `POST /api/ai-generation/tasks/{taskId}/retry` | 单任务重试接口 |
| `GET /api/ai-generation/tasks/{taskId}` | 保留现有接口 |
| `GET /api/ai-generation/tasks/latest` | 保留现有接口 |

### 6.2 任务列表接口返回字段

| 字段 | 说明 |
|---|---|
| `taskId` | 任务 ID |
| `type` | 任务类型 |
| `status` | `PENDING / PROCESSING / COMPLETED / FAILED` |
| `sourceId` | 源对象 ID |
| `targetId` | 目标对象 ID |
| `resultJson` | 结果上下文 |
| `errorMessage` | 错误信息 |
| `createdAt` | 创建时间 |
| `updatedAt` | 更新时间 |
| `completedAt` | 完成时间 |

V1 可按 `updatedAt desc` 返回最近任务列表，不强制做分页；如实现成本低，可加 `limit` 参数，默认最近 50 条。

### 6.3 扩展同步任务接入

当调用“同步当前页职位”时：

| 步骤 | 处理 |
|---|---|
| 1 | 创建 `JOB_DRAFT_PAGE_SYNC` 任务记录 |
| 2 | 任务进入 `PROCESSING` |
| 3 | 完成草稿批次创建后写入 `batchId` 到 `resultJson` |
| 4 | 成功后置为 `COMPLETED`，失败则置为 `FAILED` 并记录 `errorMessage` |

说明：V1 可先在系统接口调用链中同步完成任务记录，不强制把该任务改造成 Redis Stream 异步消费者任务；只要状态在系统中可追踪即可。

### 6.4 JD 补全任务接入

当调用草稿项/正式职位 `detail-sync` 时：

| 步骤 | 处理 |
|---|---|
| 1 | 创建 `JOB_DRAFT_DETAIL_SYNC` 任务记录 |
| 2 | 执行 JD 补全、精匹配、开场话术回写 |
| 3 | 将 `batchId` / `draftItemId` / `jobId` 等写入 `resultJson` |
| 4 | 成功/失败后更新任务状态 |

### 6.5 重试语义

| 任务类型 | 重试方式 |
|---|---|
| `RESUME_JOB_DRAFT` | 重新触发原“从简历生成职位草稿”逻辑 |
| `INTERVIEW_SESSION_CREATE` | 重新触发原“创建面试会话”逻辑 |
| `JOB_DRAFT_PAGE_SYNC` | 使用该任务 `resultJson/requestJson` 中保留的原始请求重新执行 |
| `JOB_DRAFT_DETAIL_SYNC` | 使用保留的原始请求重新执行 JD 补全 |

为支持重试，V1 需要确保任务实体中保留足够的请求上下文；若现有模型没有请求载荷字段，则在 V1 中补一个最小请求 JSON 字段，或在 `resultJson` 之外新增 `requestJson` 字段承接原始请求参数。

## 7. 前端页面设计

### 7.1 页面入口

| 项 | 设计 |
|---|---|
| 路由 | `/tasks` |
| 菜单入口 | 侧边栏新增“任务中心” |
| 访问权限 | 登录后可访问 |

### 7.2 页面结构

| 区域 | 内容 |
|---|---|
| 顶部统计卡 | 全部任务、进行中、失败、已完成 |
| 筛选区 | 按任务类型、状态筛选 |
| 列表区 | 展示任务行 |
| 行内操作 | 查看结果、重试、刷新 |

### 7.3 列表字段

| 字段 | 说明 |
|---|---|
| 任务类型 | 用户可读文案，如“职位草稿生成”“当前页同步”“JD 补全” |
| 来源对象 | 如“简历 #12”“职位草稿批次 XXX” |
| 状态 | 待处理/处理中/已完成/失败 |
| 错误信息 | 失败时展示 |
| 创建时间 | 任务发起时间 |
| 更新时间 | 最近状态更新时间 |

### 7.4 行内操作

| 操作 | 适用状态 | 行为 |
|---|---|---|
| 查看结果 | 已完成 | 跳转到对应业务页面 |
| 重试 | 失败 | 调用重试接口 |
| 刷新 | 任意 | 刷新列表 |

## 8. 结果跳转规则

| 任务类型 | 跳转目标 |
|---|---|
| `RESUME_JOB_DRAFT` | `/jobs/drafts?batchId=...` |
| `JOB_DRAFT_PAGE_SYNC` | `/jobs/drafts?batchId=...` |
| `JOB_DRAFT_DETAIL_SYNC` | 优先 `/jobs/drafts?batchId=...`；若是正式职位则跳 `/jobs` 并高亮对应 `jobId` |
| `INTERVIEW_SESSION_CREATE` | 优先跳面试记录页；若有明确会话详情入口，则跳该详情页 |

V1 若职位工作台尚未支持按 `jobId` 精准高亮，可先跳 `/jobs` 并保留后续增强空间。

## 9. 用户主流程

| 步骤 | 用户动作 | 系统行为 |
|---:|---|---|
| 1 | 用户发起职位草稿生成 / 面试生成 / 扩展同步 / JD 补全 | 系统创建任务记录 |
| 2 | 用户切换到别的页面 | 任务继续执行 |
| 3 | 用户打开任务中心 | 看到所有最近任务状态 |
| 4 | 某任务完成 | 用户点“查看结果”跳转对应页面 |
| 5 | 某任务失败 | 用户点“重试”重新执行 |

## 10. 错误处理

| 场景 | 处理 |
|---|---|
| 任务执行失败 | `status=FAILED`，展示 `errorMessage` |
| 结果上下文缺失 | “查看结果”按钮降级为不可点，并展示提示 |
| 重试失败 | 保持失败状态，错误信息更新 |
| 老任务类型无结果上下文 | V1 页面兼容展示，但不强制支持结果跳转 |

## 11. 测试与验证

| 维度 | V1 验证重点 |
|---|---|
| 后端接口 | 任务列表查询、任务重试、扩展同步任务落库、JD 补全任务落库 |
| 前端页面 | 任务列表展示、筛选、结果跳转、失败重试 |
| 兼容性 | 老的 AI 任务记录在任务中心可正常显示 |

## 12. 分阶段落地顺序

| 顺序 | 内容 |
|---:|---|
| 1 | 扩展 `AiGenerationTaskType` 与任务列表/重试接口 |
| 2 | 让“同步当前页职位”“JD 补全”创建并更新统一任务记录 |
| 3 | 新增前端“任务中心”页面、路由、菜单入口 |
| 4 | 接通结果跳转与失败重试 |

## 13. V1 完成标准

| 标准 | 说明 |
|---|---|
| 有统一任务页 | 用户可直接进入任务中心 |
| 任务来源统一 | 至少四类任务统一展示 |
| 支持切页恢复 | 离开原页面后仍可查看任务 |
| 支持失败重试 | 单任务重试可用 |
| 支持结果跳转 | 已完成任务可进入对应结果页 |
