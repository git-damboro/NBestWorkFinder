# Job 前端工作台设计文档

**日期：** `2026-04-18`  
**模块：** `frontend / job`  
**状态：** 已确认设计，待进入实现

---

## 1. 目标

为当前项目补齐 `job` 模块前端第一版，让用户可以在前端完成职位管理闭环：

1. 查看自己的职位列表
2. 新增职位并录入 JD
3. 编辑职位内容与投递状态
4. 删除职位
5. 基于已有简历执行职位匹配分析
6. 在同一个页面中查看职位详情与匹配结果

本次目标是尽快交付一个**可用、可维护、与现有页面风格一致**的单页工作台，不引入复杂多页面流转。

## 2. 范围

### 2.1 本次要做

| 功能 | 说明 |
|---|---|
| 导航入口 | 在左侧导航新增“职位管理”入口 |
| 单页路由 | 新增 `/jobs` 页面 |
| 职位列表 | 展示职位标题、公司、地点、薪资、状态、技术标签、创建时间 |
| 搜索与筛选 | 支持关键词搜索与按 `JobApplicationStatus` 状态筛选 |
| 新增职位 | 使用弹窗表单录入职位 |
| 编辑职位 | 复用新增表单，回填并更新 |
| 删除职位 | 复用现有确认对话框 |
| 详情查看 | 在主页面查看完整 JD、备注、标签、状态与时间信息 |
| 简历匹配 | 选择简历后调用匹配接口，并展示匹配结果 |

### 2.2 本次不做

| 不做项 | 原因 |
|---|---|
| 独立职位详情页 | 当前单页工作台交互更快、实现成本更低 |
| 批量操作 | 非首版核心能力 |
| 拖拽看板 | 超出当前需求 |
| 独立标签管理 | 后端已自动提取 `techTags`，前端只负责展示 |
| 高级筛选组合器 | 首版先保持简单易用 |

## 3. 已知后端契约

| 接口 | 用途 | 前端关注点 |
|---|---|---|
| `POST /api/jobs` | 新增职位 | `title / company / description` 必填 |
| `GET /api/jobs?status=` | 查询职位列表 | `status` 可选 |
| `GET /api/jobs/{id}` | 获取职位详情 | 详情比列表字段更完整 |
| `PUT /api/jobs/{id}` | 更新职位 | 只有 `description` 变化时才重新提取标签 |
| `DELETE /api/jobs/{id}` | 删除职位 | 成功后刷新列表即可 |
| `POST /api/jobs/{id}/match?resumeId=` | 简历匹配 | `resumeId` 是 query 参数，不是 body |

### 3.1 关键数据结构

| DTO | 关键字段 |
|---|---|
| `JobListItemDTO` | `id` `title` `company` `location` `salaryMin` `salaryMax` `techTags` `applicationStatus` `createdAt` |
| `JobDetailDTO` | 列表字段 + `description` `notes` `updatedAt` |
| `JobMatchDTO` | `overallScore` `matchedSkills` `missingSkills` `suggestions` `summary` |

### 3.2 状态枚举

`JobApplicationStatus`：

- `SAVED`
- `APPLIED`
- `INTERVIEWING`
- `OFFERED`
- `REJECTED`

## 4. 前端方案选择

本次采用 **A 方案：单页工作台**。

### 4.1 为什么选择单页工作台

1. 与当前开发节奏最匹配，能最快补齐职位模块可用性
2. 复用现有 `HistoryPage`、`KnowledgeBaseManagePage` 的交互模式最自然
3. 新增、编辑、删除、匹配都能在一个上下文内完成，减少跳转
4. 未来如果页面复杂度上升，仍然可以从单页平滑拆成独立详情页

### 4.2 不选多页面方案的原因

多页面虽然更“规范”，但当前项目最缺的是**可用闭环**而不是更重的页面结构。当前优先级更适合先把主功能做出来，再视复杂度决定是否拆页。

## 5. 页面结构设计

路由：`/jobs`

页面采用“**工具栏 + 列表区 + 详情区 + 对话框**”结构。

### 5.1 顶部工具栏

| 元素 | 作用 |
|---|---|
| 页面标题 | 明确这是职位管理工作台 |
| 新增职位按钮 | 打开创建职位弹窗 |
| 状态筛选 | 按投递状态筛选职位 |
| 搜索框 | 对职位标题、公司进行前端搜索 |

### 5.2 列表区

展示用户自己的职位列表，每项显示：

- 职位标题
- 公司名
- 地点
- 薪资范围
- 技术标签
- 当前状态
- 创建时间
- 操作按钮：编辑 / 删除 / 匹配

点击列表项后，在详情区展示完整信息。

### 5.3 详情区

当有选中职位时，展示：

- 标题、公司、地点、薪资
- 状态
- 技术标签
- JD 正文
- 备注
- 创建时间、更新时间

详情区同时放置常用操作：

- 编辑
- 删除
- 匹配简历

### 5.4 对话框

| 对话框 | 作用 |
|---|---|
| 职位表单弹窗 | 新增/编辑职位 |
| 删除确认弹窗 | 复用现有 `DeleteConfirmDialog` |
| 匹配弹窗 | 选择简历并展示匹配结果 |

## 6. 组件设计

### 6.1 页面与组件拆分

优先保证开发效率，同时避免 `JobManagePage.tsx` 过度膨胀。

| 文件 | 职责 |
|---|---|
| `frontend/src/pages/JobManagePage.tsx` | 页面主容器，管理列表、筛选、选中项和整体状态 |
| `frontend/src/api/jobs.ts` | 职位 API 封装 |
| `frontend/src/types/job.ts` | 前端类型定义 |
| `frontend/src/components/Layout.tsx` | 增加导航入口 |
| `frontend/src/App.tsx` | 增加 `/jobs` 路由 |
| `frontend/src/components/JobFormDialog.tsx` | 新增/编辑职位表单弹窗 |
| `frontend/src/components/JobMatchDialog.tsx` | 简历匹配弹窗与结果展示 |

### 6.2 拆分原则

- 页面负责数据流和交互组织
- 弹窗负责单一功能，不承担页面级状态
- API 与类型独立，便于后续继续扩展职位功能

## 7. 数据流设计

### 7.1 页面加载

1. 进入 `/jobs`
2. 请求职位列表 `GET /api/jobs`
3. 默认选中第一条职位，或保持未选中状态
4. 在详情区展示选中项内容

### 7.2 创建职位

1. 用户点击“新增职位”
2. 打开表单弹窗
3. 填写并提交 `POST /api/jobs`
4. 成功后刷新列表
5. 自动选中新创建职位并展示详情

### 7.3 编辑职位

1. 用户点击“编辑”
2. 用当前详情数据回填表单
3. 提交 `PUT /api/jobs/{id}`
4. 成功后刷新列表和详情

### 7.4 删除职位

1. 用户点击“删除”
2. 打开确认弹窗
3. 提交 `DELETE /api/jobs/{id}`
4. 成功后从列表移除
5. 如果删除的是当前选中项，则自动切换到下一项或清空详情区

### 7.5 简历匹配

1. 用户点击“匹配简历”
2. 弹出简历选择框
3. 使用已有简历列表接口获取简历选项
4. 选中简历后调用 `POST /api/jobs/{id}/match?resumeId=...`
5. 在匹配弹窗中展示 `overallScore`、匹配技能、缺失技能、建议和总结

## 8. 错误处理与边界情况

| 场景 | 处理方式 |
|---|---|
| 列表加载失败 | 页面展示错误提示和重试入口 |
| 没有职位 | 展示空状态，引导用户创建第一个职位 |
| 没有简历 | 匹配弹窗提示先上传简历，并给出跳转建议 |
| 表单校验失败 | 在弹窗中展示字段级错误或通用错误 |
| 删除失败 | 保持原列表状态，并显示错误提示 |
| 匹配失败 | 匹配弹窗内展示失败信息，不影响主页面 |
| 接口返回业务错误 | 复用现有 `request.ts` 错误语义，页面层只负责展示 |

特殊说明：

- 因为 `match` 接口依赖简历文本，若简历未成功解析，前端需要把错误信息直接反馈给用户
- 因为标签由后端自动生成，前端不允许手动编辑 `techTags`

## 9. 视觉与交互约束

为与现有项目保持一致，本次页面遵循以下约束：

| 约束 | 说明 |
|---|---|
| UI 风格 | 延续 `HistoryPage` 与 `KnowledgeBaseManagePage` 的卡片/表格风格 |
| 动画 | 继续使用已有 `framer-motion` 轻动画，不增加复杂转场 |
| 主题 | 保持浅色/深色模式兼容 |
| 图标 | 继续使用 `lucide-react` |
| 交互密度 | 以“信息清晰、操作直接”为主，不追求复杂布局 |

1. 先补 `types` 和 `api`
2. 再加路由和导航入口
3. 再做 `JobManagePage`
4. 再补表单弹窗
5. 再补匹配弹窗
6. 最后做最小验证与文档同步

## 12. 本设计的明确结论

| 项 | 结论 |
|---|---|
| 是否拆多页面 | 否，首版不拆 |
| 是否做职位详情页 | 否，先内嵌在工作台中 |
