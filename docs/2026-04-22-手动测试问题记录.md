## 四、BUG-004 当前证据

| 证据 | 结论 |
|---|---|
| `hs_err_pid24824.log` 显示 `There is insufficient memory for the Java Runtime Environment to continue` | 后端 JVM 曾因系统级内存/页文件不足崩溃 |
| `hs_err_pid24824.log` 显示 `Native memory allocation (mmap) failed to map 255852544 bytes` | 不是普通业务异常，而是 JVM 申请约 244MB 原生内存失败 |
| `hs_err_pid24824.log` 显示 `TotalPageFile size 29405M (AvailPageFile size 230M)` | 崩溃时 Windows 提交内存/页文件几乎耗尽 |
| 当前后端 JVM `MaxHeapSize=4257218560` | 本地开发运行未限制堆大小，默认最大堆约 4GB，内存紧张时风险较高 |
| Docker Compose 当前 `postgres`、`redis` 运行，`rustfs` 为 `unhealthy` | 依赖服务已恢复监听；RustFS 健康检查异常仍需单独处理 |
| PostgreSQL 日志出现 `database system was not properly shut down; automatic recovery in progress` | 数据库容器曾非正常停止/重启 |
| 数据库 `job_draft_batches` 最近批次 `jdb_8bfe4c6a2383` 状态为 `COMPLETED`、`imported_count=3` | 最近一次职位草稿导入已经成功落库 |
| 数据库 `jobs` 表出现导入后的 3 条职位，ID 为 5、6、7 | 导入接口本身已完成写入 |
| `JobDraftService#importItems` 只做数据库读取/写入，不调用 AI 和文件存储 | 这条导入链路理论上不会直接触发大模型或 RustFS |

## 五、BUG-004 当前判断

| 判断项 | 说明 |
|---|---|
| 当前最可能原因 | 本地系统内存/页文件耗尽导致后端 JVM 崩溃或短时不可用，前端随后显示连接失败 |
| 与导入接口的关系 | 目前证据显示“导入已成功落库”，因此报错更可能发生在导入后的 `loadBatch` 刷新或跳转加载阶段，而不是导入事务本身失败 |
| 已完成处置 | 已限制本地 `bootRun` 默认堆和元空间，并补充草稿导入开始/完成日志 |
| 尚未确认 | 当时前端具体错误文案和浏览器 Network 中失败的请求 URL/状态码；本轮按要求暂不做本地复现验证 |
| 下一步建议 | 下次允许本地复现时，用新 `bootRun` 配置重新跑职位草稿导入链路，观察是否仍断连 |
