# Java 后端面试核心知识库

## 1. Java 基础

### 1.1 面向对象三大特性

Java 面向对象的三大特性是封装、继承和多态。

- 封装：隐藏对象内部实现细节，通过公开方法控制访问，提高代码安全性和可维护性。
- 继承：子类复用父类属性和方法，适合表达稳定的“is-a”关系。
- 多态：同一接口或父类引用在运行期表现出不同实现，常见于接口编程、策略模式和模板方法。

面试回答时可以结合实际项目说明，例如在订单系统中使用统一的 `PaymentService` 接口，不同支付方式实现不同策略。

### 1.2 `equals` 和 `hashCode`

`equals` 用于判断两个对象在业务意义上是否相等，`hashCode` 用于散列容器定位桶位置。

重要原则：

1. 如果两个对象 `equals` 为 `true`，它们的 `hashCode` 必须相同。
2. 如果两个对象 `hashCode` 相同，`equals` 不一定为 `true`。
3. 放入 `HashMap`、`HashSet` 后，参与计算的字段最好不要再变化。

常见风险是只重写 `equals` 不重写 `hashCode`，会导致集合去重失败或查找异常。

### 1.3 String、StringBuilder、StringBuffer

- `String` 不可变，适合少量字符串拼接和常量场景。
- `StringBuilder` 可变、非线程安全，单线程拼接性能更好。
- `StringBuffer` 可变、线程安全，但同步开销更高。

在高频循环拼接字符串时，应优先使用 `StringBuilder`。

## 2. 集合框架

### 2.1 ArrayList 和 LinkedList

`ArrayList` 底层是动态数组，随机访问快，尾部追加效率高，但中间插入和删除可能移动大量元素。

`LinkedList` 底层是双向链表，节点插入删除更灵活，但随机访问需要遍历，缓存局部性较差。

实际开发中，大多数场景优先选择 `ArrayList`，除非确实存在频繁头部插入或队列式操作。

### 2.2 HashMap 核心机制

JDK 8 中 `HashMap` 底层结构是数组 + 链表 + 红黑树。

核心流程：

1. 根据 key 的 `hashCode` 计算 hash。
2. 根据数组长度定位桶下标。
3. 如果桶为空，直接放入。
4. 如果桶冲突，先比较 hash，再比较 key 是否相等。
5. 链表过长且数组容量达到阈值后，可能树化为红黑树。

常见问题：

- 初始容量设置不合理会导致频繁扩容。
- key 可变会导致查找失败。
- 多线程下应使用 `ConcurrentHashMap`，不要直接共享普通 `HashMap`。

## 3. 并发编程

### 3.1 线程池参数

线程池核心参数包括：

- `corePoolSize`：核心线程数。
- `maximumPoolSize`：最大线程数。
- `keepAliveTime`：非核心线程空闲存活时间。
- `workQueue`：任务队列。
- `threadFactory`：线程工厂。
- `rejectedExecutionHandler`：拒绝策略。

线程池调优需要结合任务类型：

- CPU 密集型：线程数接近 CPU 核心数。
- IO 密集型：线程数可以适当高于 CPU 核心数。

生产环境应避免使用无界队列，否则高峰期可能造成内存压力。

### 3.2 synchronized 和 ReentrantLock

`synchronized` 是 JVM 层面的内置锁，语法简单，自动释放锁。

`ReentrantLock` 是 JUC 提供的显式锁，支持可中断、超时、公平锁和多个条件队列，但必须在 `finally` 中释放。

如果只是普通互斥，优先使用 `synchronized`；如果需要更强控制能力，可使用 `ReentrantLock`。

### 3.3 volatile

`volatile` 可以保证可见性和禁止指令重排序，但不能保证复合操作的原子性。

例如 `count++` 包含读取、加一、写回三个步骤，即使用 `volatile` 也不是线程安全的。需要使用 `AtomicInteger` 或锁。

## 4. Spring Boot

### 4.1 Spring Bean 生命周期

Spring Bean 常见生命周期：

1. 实例化。
2. 属性填充。
3. Aware 接口回调。
4. BeanPostProcessor 前置处理。
5. 初始化方法。
6. BeanPostProcessor 后置处理。
7. 容器关闭时执行销毁逻辑。

理解 Bean 生命周期有助于排查依赖注入、代理增强和初始化顺序问题。

### 4.2 Spring MVC 请求流程

典型请求流程：

1. 请求进入 `DispatcherServlet`。
2. 根据 HandlerMapping 找到 Controller。
3. HandlerAdapter 执行具体方法。
4. 参数解析、校验、类型转换。
5. 调用业务 Service。
6. 返回结果并由消息转换器序列化为 JSON。

项目中常用统一响应结构和全局异常处理来保证接口格式一致。

## 5. MySQL

### 5.1 索引设计原则

常见索引原则：

- 高选择性字段适合建索引。
- 高频查询条件、排序字段、关联字段适合建索引。
- 联合索引遵循最左前缀原则。
- 避免在索引字段上进行函数计算，否则可能导致索引失效。

索引不是越多越好，过多索引会降低写入性能并增加维护成本。

### 5.2 事务隔离级别

MySQL 常见隔离级别：

- 读未提交：可能出现脏读。
- 读已提交：避免脏读，但可能不可重复读。
- 可重复读：MySQL InnoDB 默认级别，通常可避免不可重复读。
- 串行化：隔离最强，但并发性能最低。

InnoDB 通过 MVCC 和锁机制实现高并发事务控制。

## 6. Redis

### 6.1 Redis 常见数据结构

- String：缓存、计数器、分布式锁。
- Hash：对象属性存储。
- List：队列、消息列表。
- Set：去重集合。
- Sorted Set：排行榜、延迟队列。

选择数据结构时应从访问模式、数据规模和更新频率出发。

### 6.2 缓存问题

常见缓存问题包括：

- 缓存穿透：查询不存在的数据，可使用布隆过滤器或缓存空值。
- 缓存击穿：热点 key 过期瞬间大量请求打到数据库，可使用互斥锁或逻辑过期。
- 缓存雪崩：大量 key 同时过期，可设置随机过期时间。

## 7. 项目回答模板

面试回答建议使用 STAR 法则：

- Situation：项目背景。
- Task：承担任务。
- Action：采取行动。
- Result：最终结果。

例如回答性能优化问题时，可以说明接口 QPS、慢查询定位、索引优化、缓存策略和最终耗时下降比例。

4. 你的接口如何做鉴权？
5. 异步任务失败后怎么处理？
6. 文件上传如何保证安全性？
7. 你如何设计数据库表？

回答项目问题时，要避免只说“我们用了某某框架”，应该解释设计原因、权衡和结果。

- 为什么考虑这个岗位？
- 你的优势是什么？
- 遇到压力如何处理？
- 期望薪资是多少？
- 未来 1~3 年职业规划是什么？

回答时保持真实、具体、有逻辑。薪资问题可以结合市场水平、个人能力和岗位匹配度表达。
